NAME
       xxe - overview of XML External Entity vulnerability and exploitation
       techniques

DESCRIPTION
       XML External Entity (XXE) is a critical vulnerability that can allow an
       attacker to interact with data processing within an application.  This
       vulnerability can lead an attacker to access the internal filesystem of
       the application or the server, backend system, and many more critical
       resources.

       When discussing data transmission between website and server, XML and JSON
       are the two main technologies that play into existence.  An application
       that follows the process of data transmission using the standard library
       or API, but for XXE vulnerabilities there are some predefined and
       potentially dangerous functions that are enabled by default can leverage
       critical bugs which leads to a maximum data breach.

   XML Overview
       XML stands for "extensible markup language" and this language is designed
       for storing and transporting data.  Like HTML, XML follows the tree-like
       structure but this language does not support predefined tags; we have to
       create the tags according to the descriptive words.

       The XML document type definition (DTD) contains declarations that can
       define the structure of an XML document.  The DTD is declared with the
       DOCTYPE element at the start of the XML document.

       An External entity will look like:

              <!DOCTYPE foo [ <!ENTITY ext SYSTEM "file:///path/to/file" > ]>

       This feature is External Entity.  As mentioned above, DTD files are the
       special creatures that tell about the format or structure of XML data.
       The DTD file contains an element called ENTITY.

       For more details on XML entities, see:
       ⟨https://gist.github.com/RAJANAGORI/8700aa5343c03c38eb3a1dbaf13216c6/raw/22f9a976ab7797d953b929da4903563f70e03585/Entity.md⟩
       ```gist:https://gist.github.com/RAJANAGORI/8700aa5343c03c38eb3a1dbaf13216c6/raw/22f9a976ab7797d953b929da4903563f70e03585/Entity.md
       <?xml version="1.0" encoding="ISO-8859-1"?>
       <!DOCTYPE example [
       <!ELEMENT example ANY >
       <!ENTITY file SYSTEM "file:///etc/passwd" >
       ]>
       <configuration>&file;</configuration>
       ```

   Exploitation Methods
       There are three main ways to exploit XXE vulnerabilities:

       1. Exploit XXE to access the internal file system
       2. Exploit XXE to perform SSRF (Server Side Request Forgery)
       3. Exploit blind XXE to data exportation

   Internal File-system Exploitation using XXE
       This method allows an attacker to read files from the server's internal
       filesystem.

       Steps to exploit:

       1. Intercept the request (should be POST)
       2. Check the content type:
              •  If content-type is application/xml, it may be a win-win
                 situation
              •  If content-type is application/json, convert it to XML and try
                 to check the response

       Content Type Conversion
              You can change the content type using the Burp Suite extension:
              ⟨https://github.com/PortSwigger/content-type-converter⟩

       3. Insert the external entity definition between the XML declaration and
          element of the structure.  See example:
              ⟨https://gist.github.com/RAJANAGORI/719d04988a416baef93fe99605059ed7/raw/25000f5bd4cdc4ed38d4634ce689ad78186b69/external_entity.md⟩
              ```gist:https://gist.github.com/RAJANAGORI/719d04988a416baef93fe99605059ed7/raw/25000f5bd4cdc4ed38d4634ce689ad78186b69/external_entity.md
              (External entity example - gist content not available)
              ```

       4. Mention the reference of the external entity from any value to &xxe;

       5. Send the manipulated request to the repeater and check the response
          for the bad request or the 400 status code.

   SSRF Exploitation using XXE
       This method allows an attacker to perform Server Side Request Forgery
       attacks by making the server send HTTP requests to arbitrary URLs.

       In this exploitation method, you only need to change the external entity
       and replace the local file directory with the arbitrary server URL.

   Blind External Exploitation using XXE
       Blind XXE occurs when an application is vulnerable to XXE but does not
       return any defined value for the external entity in the response section.
       In general, direct getting or retrieval of the information from the
       server-side is not possible.

       Logic Behind Blind XXE
              To retrieve data in blind XXE scenarios:

              1. Define an XML parameter, say file which refers to /etc/passwd
              2. Define another entity for another XML parameter that will be
                 responsible for the HTTP request
              3. The server will make an outbound HTTP request containing the
                 file contents

NOTES
       Prevention Strategies

       •  Always use less complex data transmission standards such as JSON
          instead of XML when possible.

       •  Disable the XML External Entity and DTD processing in the application
          configuration.

       •  Implementation of server-side input validation or whitelisting, filter
          or sanitizing the data in order to prevent such attacks from
          server-side.

       •  Use a whitelist of allowed XML elements and attributes, rejecting any
          that are not explicitly permitted.

       •  Keep XML parsers and libraries up to date with the latest security
          patches.

       •  Use less complex parsers that do not support external entity
          processing.

       •  Implement Content Security Policy (CSP) and other security headers
          to mitigate potential impacts.
